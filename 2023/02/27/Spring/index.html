Spring IOC

Autowired和Resource关键字的区别?

@Autowired是Spring自己提供的注解，@Resource是Java自带的注解

@Autowired按照byType注入，如果想实现byName注入需要配合@Qualifier

@Resource默认通过byName注入，使用属性name指定bean名字，使用属性type则使用byType注入

IOC的理解

IOC是控制反转，把创建对象的主动权和时机交给Spring 容器，并通过依赖注入的方式为对象的属性赋值，IOC容器管理了整个Bean的生命周期。本质上IOC容器可以看做一个Map，我们要使用某个对象使用某个对象时，去这个容器里面拿就可以，非常方便。

Bean的生命周期

1）实例化Bean 通过BeanDefinition，使用Java反射机制，实例化Bean

2）实例化的Bean里面的属性还没有赋值，第二步是使用依赖注入的方式为Bean的属性进行赋值

3）调用Bean的前置处理器在Bean初始化前对Bean进行一些操作

4）初始化Bean

5）调用Bean的后置处理器在Bean初始化之后做一些操作（AOP就是在这个过程中，生成代理对象对原始对象对功能进行增强）

6）经过这个步骤之后，Bean就被正确创建了，创建好的Bean会放入到单例池中

如何解决循环依赖

Spring使用三级缓存来解决循环依赖。举个例子：先实例化A，在对A进行属性注入时，发现A依赖于B，在单例池中没找到B，就会去创建B，在对B进行属性注入时发现B依赖于A，A此时还没有初始化完成，放到单例池中，这样就因为循环依赖死锁了。Spring如何解决的呢，我们注意到此时A已经实例化了，我们可以将A放入到一个单独的Map中进行保存，这就是Spring的二级缓存，earlySingleObjects，专门存放那些已经创建到还没初始化完成的Bean。貌似使用二级缓存就能解决循环依赖的问题，那为什么需要三级缓存呢，这里就不得不提Spring的AOP机制。我们知道在Bean初始化之后，Bean的后置处理器可以对Bean进行增强，创建代理对象。如果直接把A放入二级缓存，B注入的就是原始A对象，AOP机制就失效了。因此我们需要引入三级缓存，三级缓存存放的是Bean的lamda表达式，通过这个表达式我们能计算出Bean是否需要提前AOP，如果需要，我们就创建Bean的AOP代理对象，存放在二级缓存，这样就完美解决了循环依赖的问题。

总结一下：在发生循环依赖时，首先会到单例池（也就是一级缓存）中寻找对象，如果不存在，就到二级缓存（earlySingleObjects）中寻找对象，如果二级缓存也不存在，就到一级缓存中，根据lamda表达式计算是否需要创建Bean到代理对象，创建完的对象会放入到二级缓存。在整个流程结束后，会将二级缓存已经初始化完成的Bean转移到一级缓存。

Spring AOP

AOP的作用

AOP(Aspect-Oriented Programming，面向切面编程)能够将那些与业务无关，却为业务模块所 共同调用的逻辑或责任(例如事务处理、日志管理、权限控制等)封装起来，便于减少系统的重复代码，降低模块间的耦合度，并有利于未来的可扩展性和可维护性。

AOP是IOC的一个扩展功能，在Bean的生命周期中，Bean初始化后，又一个后置处理器BeanPostProcessor对Bean进行增强，生成原始Bean的代理类。通常有两种代理方式，一种是JDK代理，另一种是Cglib代理。

AOP中有几个比较重要的概念，切面，切点，通知

切面可以看做切点和通知的一个集合。

切点限定了方法的生效的范围

通知定义我们方法的生效时机，分为5种

前置通知（Before）：在目标方法被调用之前调用通知功能 （参数校验、身份认证）
后置通知（After）：在目标方法完成之后调用通知，此时不会关心方法的输出是什么 （日志）
返回通知（After-returning）：在目标方法成功执行之后调用通知
异常通知（After-throwing）：在目标方法抛出异常后调用通知
环绕通知（Around）：通知包裹了被通知的方法，在被通知的方

增强后的代理类，会根据通知和切点生成拦截器链，执行通知方法。

Spring 事务

Spring事务是基于AOP来完成的。JDBC的事务流程是这样的，新建一个连接，然后关闭连接的自动提交，执行业务SQL，根据业务代码是否抛出异常来决定提交事务还是回滚事务。如果由我们自己来控制事务的提交、会滚，代码会变得很重复，耦合性很高、还容易出错。因此，Spring使用AOP机制来替我们完成这些事情，建立连接、关闭自动提交可以使用前置通知来完成，提交事务可以使用后置通知，事务回滚可以使用异常通知。Spring使用@Transactional注解来决定是否开启事务，事务的传播级别。

Spring事务的传播级别

Spring事务定义了7种传播机制:

  1. PROPAGATION_REQUIRED:默认的Spring事物传播级别，若当前存在事务，则加入该事务，若 不存在事务，则新建一个事务。

  2. PAOPAGATION_REQUIRE_NEW:若当前没有事务，则新建一个事务。若当前存在事务，则新建 一个事务，新老事务相互独立。外部事务抛出异常回滚不会影响内部事务的正常提交。

  3. PROPAGATION_NESTED:如果当前存在事务，则嵌套在当前事务中执行。如果当前没有事务， 则新建一个事务，类似于REQUIRE_NEW。

  4. PROPAGATION_SUPPORTS:支持当前事务，若当前不存在事务，以非事务的方式执行。

  5. PROPAGATION_NOT_SUPPORTED:以非事务的方式执行，若当前存在事务，则把当前事务挂

    起。

  6. PROPAGATION_MANDATORY:强制事务执行，若当前不存在事务，则抛出异常.

  7. PROPAGATION_NEVER:以非事务的方式执行，如果当前存在事务，则抛出异常。

SpringMVC

DispatchServlet处理流程

  • 用户发送请求至前端控制器DispatcherServlet。

  • DispatcherServlet收到请求调用HandlerMapping处理器映射器。

  • 处理器映射器找到具体的处理器(可以根据xml配置、注解进行查找)，生成处理器对象及处理器 拦截器(如果有则生成)一并返回给DispatcherServlet。

  • DispatcherServlet调用HandlerAdapter处理器适配器。

  • HandlerAdapter经过适配调用具体的处理器(Controller，也叫后端控制器)。

  • Controller执行完成返回ModelAndView。

  • HandlerAdapter将controller执行结果ModelAndView返回给DispatcherServlet。

  • DispatcherServlet将ModelAndView传给ViewReslover视图解析器。

  • ViewReslover解析后返回具体View。

  • DispatcherServlet根据View进行渲染视图(即将模型数据填充至视图中)。

  • DispatcherServlet响应用户。

SpringBoot

为什么要用Springboot

1、独立运行，Springboot内置了各种Servelt容器，比如tomcat、Jetty，不需要打成war包部署到容器中运行，只需要打一个jar包就能独立运行。

2、自动装配 Springboot到自动装配机制，允许我们快速地整合各种其他框架如Mytatis等，通常我们只需要进行简单的配置。

启动流程

自动装配原理

自动装配目的是快速地整合第三方组件，将第三方组件的Bean自动加载到IOC容器中来。其实现依赖于@SpringBootApplication注解，这是个复合注解，真正实现自动装配的注解是@EnbaleAutoConfiguration，这个注解会导入一个AutoConfigurationImportSelector类。

这个类会通过spring.factories文件获取到所有需要进行自动装配的配置类，通常我们整合或者自己定义

自动配置简单来说呢，就是将第三方的组件自动装载到IOC容器里面，不需要开发人员再去编写相关的配置，在SpringBoot应用里面呢只需要加上@SpringBootApplication注解就可以实现自动配置，SpringBootApplication它是一个复合注解，真正实现自动装配的注解是@EnableAutoConfiguration注解。自动装配的实现呢主要依靠三个核心的关键技术：
1）引入Starter，启动依赖组件的时候，这个组件里面必须包括@Configuration配置类，然后我需要通过Bean注解去声明需要装配到IOC容器里面的Bean对象
2）这个配置类是放在第三方的jar包里面，然后通过Spring Boot中约定大于配置的理念，去把配置类的全路径放在件META_INF/Spring.factories文件里面，SpringBoot就可以知道第三方jar包里面配置类的位置，它主要是依靠Spring里面的SpringFactorierLoader来完成的
3）SpringBoot拿到所有第三方jar包声明的配置类之后，再通过ImportSelector这样一个接口来实现对这些配置类的动态加载，从而去完成自动装配这样的一个动作。
在我看来，SpringBoot是约定优于配置这一理念下的一个产物，所以在很多的地方都会看到这一类的思想，它的出现让开发人员可以更加聚焦在业务代码的编写上，而不需要去关系和业务无关的配置
