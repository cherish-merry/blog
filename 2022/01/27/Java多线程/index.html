2 重要方法

方法名 功能说明 注意
start() 启动一个新线程，在新的线程 运行 run 方法中的代码 start 方法只是让线程进入就绪，里面代码不一定立刻 运行（CPU 的时间片还没分给它）。每个线程对象的 start方法只能调用一次，如果调用了多次会出现 IllegalThreadStateException
join() 阻塞调用线程直至线程结束
yield() 让出cpu调度，线程状态由运行变为就绪 下次可能还是调度改线程
interrupt 打断线程 如果被打断线程正在 sleep，wait，join 会导致被打断 的线程抛出 InterruptedException，并清除打断标记 ；如果打断的正在运行的线程，则会设置 打断标 记 ；park 的线程被打断，也会设置 打断标记
interrupted() 判断当前线程是 否被打断 会清除 打断标记

  1. 线程池中刚开始没有线程，当一个任务提交给线程池后，线程池会创建一个新线程来执行任务。

4 JMM 内存模型

java内存模型规定所有变量保存在主内存中，每个线程有自己的工作内存。线程的工作内存中保存了该线程使用到的变量的主内存拷贝，线程对变量的所有操作都必须在工作内存中进行，而不能直接读写主内存中的变量。不同的线程之间无法直接访问对方的工作内存，需通过主内存才能进行变量的传递。

5 可见性、有序性、原子性、volatile、cas

可见性

可见性指的是当一个线程修改了共享变量后，其他线程能够立即得知这个修改，volatile可以保证可见性

有序性

有序性指的是程序按照代码的先后顺序执行

为了性能优化，编译器和处理器会可能进行指令重排序，有时候会改变程序中语句的先后顺序

原子性

原子性指的是一个或者多个操作在 CPU 执行的过程中不被中断的特性

在多线程情况下，线程会被操作系统调度进行任务切换，占有CPU时间片段的就执行，否则就阻塞

volatile

volatile 保证了共享变量的可见性和有序性，volatile 的底层实现原理是内存屏障，Memory Barrier``（Memory Fence）

对 volatile 变量的写指令后会加入写屏障，写屏障（sfence）保证在该屏障之前的，对共享变量的改动，都同步到主存当中，不会将写屏障之前的代码排在写屏障之后

对 volatile 变量的读指令前会加入读屏障，读屏障（lfence）保证在该屏障之后，对共享变量的读取，加载的是主存中最新数据，不会将读屏障之后的代码排在读屏障之前

cas、原子类

compareAndSet操作系统指令级别保证了操作的原子性，通常配合volatile使用，通过循环cas来保证操作最终完成。

各种原子类

AtomicInteger、AtomicBoolean、AtomicLong

ABA问题

AtomicStampedRefence（加上版本号）

6 对象头

每个普通对象都有一个8个字节的对象头，由Markwork和Klass word两部分组成

Markword中存储对象垃圾回收、加锁相关信息，Klass word指向类对象

7 synchronized 原理

7.1 monitor 原理

每个 Java 对象都可以关联一个 Monitor 对象，如果使用 synchronized 给对象上锁（重量级）之后，该对象头的 Mark Word 中就被设置指向 Monitor 对象的指针

Monitor 结构

刚开始 Monitor 中 Owner 为 null

当 Thread-2 执行 synchronized(obj) 就会将 Monitor 的所有者 Owner 置为 Thread-2，Monitor中只能有一 个 Owner

在Thread-2 上锁的过程中，如果 Thread-3，Thread-4，Thread-5 也来执行 synchronized(obj)，就会进入 EntryList BLOCKED

Thread-2 执行完同步代码块的内容，然后唤醒 EntryList 中等待的线程来竞争锁，竞争的时是非公平的

图中 WaitSet 中的 Thread-0，Thread-1 是之前获得过锁，但条件不满足进入 WAITING 状态的线程，后面讲 wait-notify 时会分析

7.2 轻量级锁

如果一个对象虽然有多线程要加锁，但加锁的时间是错开的（也就是没有竞争），那么可以 使用轻量级锁来优化。

在线程的栈桢中创建锁记录（Lock Record）对象，让锁记录中 Object reference 指向锁对象，并尝试用 cas 替换 Object 的 Mark Word，将Mark Word 的值存 入锁记录

如果 cas 替换成功

对象头中存储了锁记录地址和状态 00 ，表示由该线程给对象加锁，这时图示如下

如果 cas 失败，有两种情况

如果是其它线程已经持有了该 Object 的轻量级锁，这时表明有竞争，进入锁膨胀过程

如果是自己执行了 synchronized 锁重入，那么再添加一条 Lock Record 作为重入的计数

当退出 synchronized 代码块（解锁时）

如果有取值为 null 的锁记录，表示有重入，这时重置锁记录，表示重 入计数减一

7.3 偏向锁

如果开启了偏向锁（默认开启），那么对象创建后，markword 最后 3 位为 101，否则为001

可以添加VM 参数 -XX:-UseBiasedLocking 禁用偏向锁

注意

处于偏向锁的对象解锁后，线程 id 仍存储于对象头中

正常状态对象一开始是没有 hashCode 的，第一次调用才生成

调用hashcode()方法，会导致偏向锁被撤销(存储hashcode)

其他线程使用对象，会导致偏向锁被撤销（轻量级锁）

调用wait/notify，会导致偏向锁被撤销（重量级锁）

轻量级锁会在锁记录中记录 hashCode

重量级锁会在 Monitor 中记录 hashCode

7.4 synchronized 锁升级

  1. 如果开启了偏向锁（默认开启），最后三位是101，处于可偏向状态，Mark Word记录线程Id

  2. 当线程A获取锁时，会对比Mark Word中的线程id和自己的线程id。

    如果相同，获取锁成功

    如果不同，他会检查Mark Word 线程id对应的线程是否存活

    ​ 如果存活，取消偏向状态，升级为轻量级锁

    ​ 如果不存活，使用cas替换Mark Word中的线程id为自己的线程id

  3. 偏向锁状态抢锁失败，代表当前锁有一定的竞争，偏向锁将升级为轻量级锁。JVM会在当前线程的线程栈中创建一条锁记录，锁记录的Object 指针指向锁对象，使用cas操作保存Mark Word的信息，同时使用cas 操作使Mark Word 指向锁记录，开启自旋，重复尝试。

  4. 当替换失败达到一定次数时(默认是10)，就会进入锁膨胀，升级为重量级锁。如果在此期间还有其他线程竞争锁，立即升级为重量级锁。

  5. 升级重量级锁， 首先申请一个Monitor对象，让Mark Word指向Monitor 对象，把加锁失败的线程放入到Monitor 对象的EntryList中阻塞

8 wait、notify、notifyAll、join原理

wait、notify、notifyAll

当Owner 线程发现条件不满足时，调用wait 方法进入到 Monitor 的 waitSet 进行等待
当调用notify 或 notifyAll 会唤醒waitSet 中等待的线程进入entryList 竞争锁

t1.join() 将自己加入到 t1 的waitSet 直到 t1 结束

9 park unPark 原理

每个线程都有自己的一个 Parker 对象，由三部分组成 _counter ， _cond 和 _mutex

  1. 当前线程调用 Unsafe.park() 方法
  2. 检查 _counter ，本情况为 0，这时，获得 _mutex 互斥锁
  3. 线程进入 _cond 条件变量阻塞
  4. 设置 _counter = 0

  1. 调用 Unsafe.unpark(Thread_0) 方法，设置 _counter 为 1
  2. 唤醒 _cond 条件变量中的 Thread_0
  3. Thread_0 恢复运行
  4. 设置 _counter 为 0

10 AQS

10.1 简介

AQS 全称 AbstractQueuedSynchronizer，JUC 下多个同步工具，比如ReentrantLock，CountDownLatch，Semaphore 都是基于此实现

10.2 原理

AQS 是一个抽象队列式同步器，底层维护了一个state变量，一个CLH的等待队列，以及可能存在的多个
Condition队列，AQS使用LockSupport来阻塞和唤醒线程

exclusiveOwnerThread

独占模式下的同步器的线程拥有者

state

state 表示资源的状态（分独占模式和共享模式），子类需要定义如何维护这个状态，控制如何获取锁和释放锁

CLH队列

基于 FIFO的等待队列，类似于 Monitor 的 EntryList

Condition队列

条件变量来实现等待、唤醒机制，支持多个条件变量，类似于 Monitor 的 WaitSet

10.3 ReentrantLock(非公平锁) 原理

ReentrantLock有公平和非公平实现，默认使用非公平实现

lock()流程

1: CAS设置state为1，设置成功，设置占用线程为当前线程

公平锁

设置state之前，判断CLH队列中是否有线程节点，没有时才去设置state

2:不成功，查看是否占有线程为当前线程，如果是，state += 1， 代表可重入次数

3: 如果占有线程不是当前线程，将当前线程封装成一个Node节点，加入CLH队列，使用LockSupport阻塞当前线程

release()流程

1:判断占有线程是否是当前线程，如果不是，抛出异常

2:将state-=1

3:判断state是否为0，如果为0，将占用线程设置为null，唤醒CLH队列中的第一个节点线程

await()流程

1: 将自己加入到ConditionObject的等待队列中，释放锁，阻塞该线程，唤醒CLH中第一个线程节点

signal()流程

1:从ConditionObject等待队列中移除第一个节点到CLH队列中

10.4 ReentrantReadWriteLock原理

读写锁用的是同一个 Sycn 同步器，因此等待队列、state 等也是同一个，实现了读读并发

w.lock

流程与 ReentrantLock 加锁相比没有特殊之处，不同是写锁状态占了 state 的低 16 位，而读锁使用的是 state 的高 16 位

r.lock

加锁失败后，进入队列，不同于写锁的是节点类型为shared

w.unlock()

唤醒后续节点，如果后续节点为shared节点，继续唤醒后面类型为shared的节点

r.unlock()

tryReleaseShared()如果读锁state状态为0，唤醒后序节点

10.5：Semaphore原理

信号量，用来限制同时访问共享资源的线程上限

构造方法指定state的值，state > 0获取成功，否则获取失败，进入CLH队列阻塞

释放后，state + 1，唤醒后继节点

10.6 CountdownLatch原理

用来同步多个线程之间的工作，基于共享锁

设置state为初始值

await()判断state==0，如果不是，在CLH队列上阻塞。

countDown()，使state--

如果state为0，依次唤醒后面阻塞的节点类型为shared的线程

如果不是调用reentrantLock的condition的await()方法阻塞

12 ThreadLocal

ThreadLocal 中填充的的是当前线程的变量，该变量对其他线程而言是封闭且隔离的，ThreadLocal 为变量在每个线程中创建了一个副本，这样每个线程都可以访问自己内部的副本变量。

每个Thread对象底层有一个ThreadLocalMap，调用ThreadLocal的set方法时会把ThreadLocal作为键，value作为值，存储到当前线程的ThreadLocalMap中，由于ThreadLocalMap是线程私有的，对其他线程而言是封闭且隔离的。

这样有一个问题呀，如何保证ThreadLocal对象被正确回收？

当把ThreadLocal置为空时，是如何保证ThreadLocalMap里面的ThreadLocal对象被回收的？

使用虚引用，我们知道，只要发生GC时，虚引用指向的对象就会被垃圾回收，存放在ThreadLocalMap里面的ThreadLocal对象继承了WeakRefence，

当ThreadLocalMap里面的ThreadLocal被回收后，Key为空，值所指向的内存仍然没有被回收。虽然每次调用get、set会检测ThreadLocalMap里面的空键，但可能长时间不掉用，仍然有内存泄漏的风险。因此，最好记得掉用remove方法手动移除。

线程池，线程执行完后记得清理ThreadLocalMap，防止对后面的任务造成影响。
